Yarn环境配置

1）增加ApplicationMaster资源比例

容量调度器对每个资源队列中同时运行的Application Master占用的资源进行了限制，该限制通过yarn.scheduler.capacity.maximum-am-resource-percent参数实现，其默认值是0.1，表示每个资源队列上Application Master最多可使用的资源为该队列总资源的10%，目的是防止大部分资源都被Application Master占用，而导致Map/Reduce Task无法执行。

生产环境该参数可使用默认值。但学习环境，集群资源总数很少，如果只分配10%的资源给Application Master，则可能出现，同一时刻只能运行一个Job的情况，因为一个Application Master使用的资源就可能已经达到10%的上限了。故此处可将该值适当调大。

CTE

在 SQL 中，CTE（Common Table Expression，公共表表达式）是一种临时的命名结果集，可以在单个查询中定义和引用。它通常用于提高查询的可读性、模块化复杂查询或递归查询。CTE 通过 WITH 关键字定义，并且在定义它的查询范围内有效。

下面是对 CTE 的详细解释：

CTE 的基本概念

  • 定义: CTE 是一个临时的结果集，类似于子查询，但它被定义在查询的开头，并可以通过名称重复引用。
  • 作用:
    • 提高代码可读性，避免嵌套过深的子查询。
    • 支持递归查询（例如处理层次结构或树形数据）。
    • 可以被多次引用，而无需重复编写相同的逻辑。
  • 生命周期: CTE 只在定义它的查询执行期间存在，查询结束后自动销毁。

基本语法

  • cte_name: CTE 的名称，供后续查询引用。
  • column_name: 可选，指定 CTE 返回的列名。如果不指定，则使用内部查询的列名。
  • AS (...): 定义 CTE 的查询逻辑。
  • 主查询中可以通过 cte_name 像表一样使用 CTE。

CTE 的特点

  1. 可读性强:

    • CTE 将复杂逻辑拆分为命名块，比嵌套子查询更容易理解。

  2. 可多次引用:

CTE vs 子查询

特性 CTE 子查询
定义位置 查询开头，使用 WITH 嵌套在主查询中
可读性 高，逻辑分离清晰 低，嵌套多时复杂
可重复引用 是 否（需重复编写）
支持递归 是 否

使用场景

  1. 简化复杂查询: 将查询分解为多个逻辑部分。
  2. 递归查询: 处理树形结构、图数据（如组织层级、文件目录）。
  3. 数据预处理: 在主查询前准备中间结果。
  4. 调试: 单独测试 CTE 的结果。

注意事项

  • 性能: CTE 的性能通常与等效的子查询相似，具体取决于数据库优化器。某些数据库（如 PostgreSQL）会物化 CTE（缓存结果），而其他数据库（如 SQL Server）可能不会。
  • 命名唯一: 在同一查询中，多个 CTE 的名称不能重复。
  • 作用域: CTE 仅在定义它的查询中有效，不能跨查询使用。

hive执行报错

问题出现原因：

这个错误是因为hive的map join参数默认是开启的：

使用hive进行map join时， 节点内存不够就会报该类型错误

MapJoin是指在Map 端进行join，其原理是broadcast join，即把小表作为一个完整的驱动表来进行join操作。通常情况下，要连接的各个表里面的数据会分布在不同的Map中进行处理。即同一个Key对应的Value可能存在不同的Map中。这样就必须等到 Reduce中去连接。要使MapJoin能够顺利进行，那就必须满足这样的条件：除了一份表的数据分布在不同的Map中外，其他连接的表的数据必须在每个Map中有完整的拷贝。Map Join会把小表全部读入内存中，在Map阶段直接拿另外一个表的数据和内存中表数据做匹配 (这时可以使用Distributed Cache将小表分发到各个节点上，以供Mapper加载使用)，由于在map时进行了join操作，省去了reduce运行的效率也会高很多。

当机器内存不足时，无法在Map端进行join，即会报错
